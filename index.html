Integration Design

extra_config 컬럼으로
서비스별 요구사항을 흡수했습니다.

서비스마다 필요한 데이터가 달랐습니다. Slack은 bot_token/channel_id/team_id, Calendar는 access/refresh + calendar 정보, Jira는 cloud_id, site_url, project_key 등. extra_config(JSON) 컬럼으로 공통 토큰 컬럼과 서비스별 메타데이터를 분리했습니다. SQLAlchemy Mapped 타입으로 마이그레이션 안정성도 확보했습니다.

Troubleshooting

기상청 단기예보 API의
격자좌표와 트래픽 병목

단기예보 API는 위경도가 아닌 격자좌표를 사용합니다. 카카오 맵 API로 위경도 변환 후 수학 공식으로 격자좌표 매핑. 전국 수천 개 데이터 대신 12개 주요 도시(서울·인천·수원·춘천·강릉·대전·청주·광주·전주·대구·부산·제주)로 DB 저장을 최소화했습니다.

00시 호출 시 간헐적 타임아웃이 발생. 타임아웃 연장 · 최대 3회 재시도 · 호출 시간 분산을 적용했으나 여전히 실패했습니다. 이후 진짜 원인을 발견했습니다. 기상청 API 장애가 아닌 SQLAlchemy 커넥션 풀 문제였습니다. 컨테이너 장시간 실행 중 MySQL이 idle 커넥션을 강제 종료하고, 스케줄러가 죽은 커넥션으로 실행 시도해 실패한 것입니다. 커넥션을 1시간마다 자동 갱신하는 코드를 작성해 최종 해결했습니다.

결제, 재고 차감, 기록 생성이 개별 처리되면 중간 서버 다운 시 데이터 불일치가 발생합니다. 5단계를 단일 트랜잭션으로 묶어 실패 시 롤백되도록 처리.

def create_order(db: Session, user_id: int, items: List[Dict]):
    try:
        total_price = sum(item['price'] * item['quantity'] for item in items)

        order_sql = """INSERT INTO orders (user_id, total_price) VALUES (:uid, :total)"""
        result = db.execute(text(order_sql), {"uid": user_id, "total": total_price})
        order_id = result.lastrowid

        for item in items:
            db.execute(text("INSERT INTO order_items ..."), {...})
            db.execute(text("UPDATE products SET stock = stock - :qty ..."), {...})

        db.commit()
        return order_id
    except Exception as e:
        db.rollback()
        raise e

Troubleshooting

챗봇 환각 — FAQ JSON 50개로 가이드레일

OpenAI API 챗봇이 사이트와 무관한 질문에도 답하는 환각 현상이 발생. 예상 질문/답변 50개를 JSON으로 구조화하여 시스템 프롬프트에 주입, 외 질문은 거절하도록 가이드레일을 설계했습니다.

장점

별도 DB 없이 처리

FAQ JSON 주입만으로 효율적 가이드레일 구현. 응답 시간 빠름.

한계

토큰 비용 증가

FAQ가 늘수록 매 요청 토큰 비용 증가. 다음에는 DB + RAG 기반으로 전환할 계획.

Retrospective

LLM 챗봇 구현 자체는 어렵지 않았지만, FAQ 주입 방식은 예상 질문을 일일이 생각하며 하드코딩해야 해 한계가 분명했습니다. 좀 더 효율적인 RAG 방식의 필요성을 느꼈고, 이후 곰팡이 진단 앱의 ChromaDB RAG로 이어졌습니다.

파일 하나에 1000줄이 넘어가는 구조에서 불필요·중복 코드가 많이 생긴다는 걸 체감했고, 다음 프로젝트에서 DDD 아키텍처를 도입한 직접적인 계기가 되었습니다.
